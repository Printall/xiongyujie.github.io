Linux 的发行版说简单点就是将 Linux 内核与应用软件做一个打包。

启动过程

linux启动过程：内核引导、运行init、系统初始化、建立终端、用户登录

init的程序类型：

  1. ​ SysV CentOS5以前，配置文件 ：/etc/inittab
  2. ​ Upstart CenttOS6，配置文件：/etc/initab、/etc/init/*.conf
  3. ​ Systemd Centos 7 配置文件：/usr/lib/systemd/system、/etc/system/system

内核引导：打开电源开机自检，安装BIOS设置启动，操作系统接管硬件，读取/boot内容

运行init，读取/etc/inittab

运行级别：运行级别存于/etc/rc[0123456].d

  1. 运行级别0:系统停机状态，系统默认运行级别不能设为0，否则不能正常启动
  2. 运行级别1:单用户工作状态，root权限，用于系统维护，禁止远程登陆
  3. 运行级别2:多用户状态(没有NFS)
  4. 运行级别3:完全的多用户状态(有NFS)，登陆后进入控制台命令行模式
  5. 运行级别4:系统未使用，保留
  6. 运行级别5:X11控制台，登陆后进入图形GUI模式
  7. 运行级别6:系统正常关闭并重启，默认运行级别不能设为6，否则不能正常启动

系统初始化：它主要完成的工作有：激活交换分区，检查磁盘，加载硬件模块以及其它一些需要优先执行任务。不过我在unbuntu20.4没有找到inittab

建立终端，用户登录、图形模式和文字模式切换。

linux关机：

​ sync将数据同步到硬盘

​ 使用shutdown关机，showdown -[option] [param] [message]

​ option参数：-h，几分钟后关机，-r，几分钟后重启

​ reboot 重启命令，

​ halt关闭系统等于poweroff

linux的目录结构：

​

文件夹 功能
/bin 存放经常使用的命令
/boot 存放启动linux使用的核心文件
/dev 存放linux访问的设备和文件系统
/etc 存放所有系统管理需要的配置文件
/home 用户的工作空间
/lib* 系统最基本的动态链接共享库
/media 系统把识别的驱动光盘挂在下下面
/mnt 用户临时挂载的光盘的目录
/opt 主机额外安装软件摆放的目录
/proc 虚拟目录，系统内存的映射文件，这个目录的内容存放在内存中。
/root 系统管理员的主目录
/sbin 存放系统管理员使用的系统管理程序
/srv 存放服务器启动之后需要提取的文件
/sys 存放系统的设备结构，总线结构，设备块，文件系统。
/tmp 存放临时文件
/usr unix shared resource，存放用户应用程序和文件
/var 将经常修改的文件存放这个目录下
/run 临时文件系统，存储启动到现在的信息。重启之后就会清除。

物理空间

  1. 0x00000000 - 0x000F0000 ：由BIOS占据，BIOS主要功能为系统初始化，然后从硬盘加载操作系统到内存，并且将系统的操作权限给操作系统。

  2. 当前计算机还保留1MB的低地址空间方便计算机向后兼容，现在还有很多计算机还区分常规存储和扩展存储，除此之外，BIOS将最顶层的空间给PCI设备使用。

  3. JOS会使用256M的物理内存，所以接下来认为PC只有32物理内存。

BIOS的只读存储

这一部分主要是使用QEMU的调试工具研究计算机的启动方式

步骤：打开两个终端，分别进入lab文件夹，一个窗口输入make qemu-gdb，另一个输入make gdb

就可以得到如下界面

这个表中我们可以看到这样一行数据：

[f000:fff0] 0xffff0: ljmp $0x3630,$0xf000e05b

上面这一行为gdb执行的第一条指令，我们可以得到这样一些信息

  • IBM PC开始的物理地址为 0x0000fff0，这是为ROM BIOS保留的64KM的顶部
  • PC开始执行的CS = 0xf000 和 IP=0xfff0. PS:CS代码段寄存器，IP为指令指针寄存器
  • 执行的第一条指令为jmp指令

这个设计主要的希望电脑开机的时候，BIOS始终首先控制计算机。QEMU模拟器有自己的BIOS,BIOS被放置在模拟的物理地址空间中。

在程序重置的时候，模拟器进入真实模式并且设置CS和IP，所以执行开始于（CS:IP）。

但是出现一个问题如何从段地址到真实物理地址呢。

回答这个问题我们需要了解一个真实地址相关的知识。在实地址模式下也就是PC的启动模式，地址转换过程通过这样的一个公式，物理地址 = 16*segment + offset。所以就可得到先前的地址0xffff0 = 16*0xf000 + 0xfff0;

0xffff0是BIOS结束前的16个字节的地址，BISO做的第一件事就是使用jmp指令跳到较早的位置。

第二部分：引导加载程序

PC挂载的硬盘和软盘分为512个字节，被称为扇区。

扇区是一个硬盘最小的传输粒度：每次读写必须为一个或者多余一个扇区.

如果磁盘是可以引导的,则被称为引导扇区,这是引导加载程序的代码所在.

当BIOS返现一个可引导的磁盘或者硬盘,它将会加载一个扇区的内容到物理地址为0x7c00-0x7dff中,并且使用jmp指令跳到CS:IP 0000:7c00, 通过这种方法把系统的控制权限给系统.

和BIOS加载地址一样,这些地址是任意的,但是对于PC来说他们是固定的和标准的.

6.828加载过程

对于6.828来说,使用传统的硬盘，这意味着该系统引导加载程序只有512字节，引导加载程序由boot/boot.s和boot/main.c组成。引导加载程序主要由两个功能：

  1. 加载程序能够将实模式转换为32的保护模式，只有在保护模式下，软件才能获取到内存容量在1MB以上的物理空间，在保护模式下，分段地址转换为物理地址方式和实模式不同，而且转换之后的地址变为32位
  2. 引导加载程序通过X86特殊指令直接访问磁盘设备，从而直接读取内核。

obj/boot/boot.asm文件在编译引导加载程序后创建的引导加载的反汇编。这个反汇编程序可以让我们准确的看到所有引导加载程序在物理内存中的位置，然后跟踪GDB中逐步引导加载程序时发生的事情。

我们同样可以在GDB中设置断电，使用b，例如 b *0x7c00在地址0x7c00设置断点，使用c和si，c的含义为一直执行指导遇到断点，si n为一次跳过n个指令。

检查内存中的指令，使用x/i指令，该命令的语法为 x/Ni ADDR，N为反汇编连续指令数，addr是要开始反汇编的内存地址。

由上图可以观察到，当加载全局描述符表格寄存器完成后，内核开始由实模式转到保护模式。这是为了与早期的的PC兼容，默认情况下地址超过1MB的内容为零，但是转换为保护模式下，就可以使用1MB以后的内容了。

在boot.asm文件中可以看见第一条指令为cli，这个指令的目的是为了让系统加载内核的过程中拒绝中断。
